    PRD: Sample Weights for Context-Aware Statistical Tests¶

    Status: Draft
    Author: TBD
    Date: 2026-04-19

    1. Executive Summary¶

    Problem Statement¶

    The statistical tests in samesame treat every sample with equal weight, making them blind to the structure of covariate shift between train and test distributions. When deployment data differs systematically from training data, equal-weight tests can be under-powered in relevant regions and over-powered in irrelevant ones — producing misleading conclusions about model degradation.

    Proposed Solution¶

    Add an optional sample_weight parameter to the existing test classes (CTST, WeightedAUC, and all downstream tests) so that practitioners can supply externally-computed density-ratio weights. The full test pipeline — test statistics, permutation null distributions, Bayesian posteriors, p-values, and Bayes factors — must honour these weights, making every test context-aware with zero breaking changes to the current API.

    Success Criteria¶

    1. Backwards-compatibility: When sample_weight=None (default), all tests produce results numerically identical to the current unweighted implementation.
    2. Full-pipeline coverage: sample_weight flows through permutation statistics, Bayesian bootstrap posteriors, and all derived quantities (p-values, Bayes factors, CIs) without silent dropping.
    3. All three weighting scenarios supported: Scenario 1 (∝ Pr_test), Scenario 2 (∝ Pr_train), Scenario 3 (∝ Pr_test / Pr_train) are all expressible by passing the appropriate pre-computed weight array.
    4. Zero new required user dependencies: density ratio estimation is the caller's responsibility; the package ships no new mandatory dependencies.
    5. 100% test coverage on new weight-handling code paths.

    2. User Experience & Functionality¶

    User Personas¶

    Persona Description
    ML Practitioner Monitors a deployed model for data/covariate drift; wants test results that reflect the actual deployment distribution, not the historical training distribution.
    Data Scientist Runs controlled experiments (A/B tests, non-inferiority tests) where a known shift exists and equal-weight tests would be confounded.
    Applied Researcher Studies domain adaptation; needs the package to support arbitrary density-ratio reweighting schemes as a building block for more complex analyses.

    Weighting Scenarios¶

    Scenario Weight ∝ Interpretation Example Use
    1 \(\Pr_{\text{test}}(x)\) Focus on OOD / deployment-distribution samples Evaluate model where new data lives
    2 \(\Pr_{\text{train}}(x)\) Focus on in-distribution samples Guard against drift by anchoring to training support
    3 (primary) \(\Pr_{\text{test}}(x) / \Pr_{\text{train}}(x)\) Focus on common support (covariate balance) Fair comparison where both distributions overlap; density ratio ≈ 1

    User Stories¶

    Story 1 — Covariate-balance test (Scenario 3, primary)
    As an ML practitioner, I want to pass covariate-balance weights (∝ density ratio) to the adverse-shift test so that the test focuses on the overlapping support region and avoids false alarms caused by distributional extrapolation.

    Acceptance Criteria: - WeightedAUC(actual, predicted, sample_weight=w).pvalue returns a float when w sums to len(w). - When w = np.ones(n), result matches WeightedAUC(actual, predicted).pvalue to within floating-point tolerance. - Invalid weights (negative values, wrong length, all-zero) raise ValueError with a descriptive message.

    Story 2 — OOD-focused test (Scenario 1)
    As a data scientist, I want to upweight test-distribution samples so that the test is sensitive to degradation where new data actually lives.

    Acceptance Criteria: - Supplying weights ∝ Pr_test shifts power towards regions of high test-distribution density. - The CTST base class propagates weights into the permutation statistic correctly.

    Story 3 — In-distribution-focused test (Scenario 2)
    As a researcher, I want to upweight training-distribution samples so that I can isolate whether degradation is driven by the model's behaviour on familiar inputs.

    Acceptance Criteria: - Supplying weights ∝ Pr_train is expressible without any API change; it is simply a different array passed to sample_weight.

    Story 4 — Bayesian path
    As an ML practitioner, I want the Bayesian posterior (WAUC) and derived Bayes factor to also respect sample weights so that my Bayesian and frequentist analyses are consistent.

    Acceptance Criteria: - WeightedAUC(..., sample_weight=w).posterior returns a posterior array reflecting w. - WeightedAUC(..., sample_weight=w).bayes_factor is derived from the weighted posterior.

    Non-Goals¶

    • Density ratio estimation: the package will NOT estimate \(\hat{r}(x)\) or \(\Pr_{\text{test}}\). Weight computation is entirely the caller's responsibility.
    • New public estimator classes (e.g. DensityRatioEstimator): out of scope for this release.
    • Normalisation strategy selection: weights are normalised internally to sum to n_samples; no alternative normalisation schemes are exposed.
    • Weighted versions of ood.py functions (logit_gap, max_logit): these are score-extraction utilities, not tests, and are out of scope.
    • Streaming or online weight updates: batch use only.

    Component Changes¶

    samesame/ctst.py — CTST dataclass¶

    • Add field: sample_weight: NDArray | None = None
    • Add __post_init__ step: call _validate_and_normalise_weights(self.sample_weight, len(self.actual))
    • Propagate sample_weight into the statistic closure passed to scipy.stats.permutation_test.
      Critical: weights must be indexed by the original sample position, not permuted with labels. The permutation shuffles actual labels; sample_weight[i] always corresponds to observation i.

    samesame/nit.py — WeightedAUC dataclass¶

    • Inherits sample_weight from CTST with no additional fields needed.
    • bayesian_posterior call in the posterior cached property must pass sample_weight as the base weight, combined with the Bayesian bootstrap Dirichlet weights:
      w_combined = sample_weight * dirichlet_weight (element-wise), then re-normalised.

    samesame/_bayesboot.py — bayesian_posterior¶

    • Add parameter: base_weight: NDArray | None = None
    • When provided, multiply Dirichlet weights element-wise with base_weight before passing to metric:
      w = base_weight * dirichlet_w; w = w / w.sum() * n

    samesame/_utils.py — new validation helper¶

    def validate_and_normalise_weights(
    sample_weight: NDArray | None,
    n: int,
) -> NDArray | None:
    """Validate and normalise sample weights to sum to n."""

    Raises ValueError if: - len(sample_weight) != n - Any weight is negative - All weights are zero

    Returns None if input is None (pass-through, meaning equal weights).

    Integration Points¶

    Component Change Notes
    CTST Add sample_weight field + validation default=None preserves backwards-compat
    WeightedAUC Inherited; propagate to bayesian_posterior
    _bayesboot.bayesian_posterior Add base_weight param Combined with Dirichlet draw
    _utils Add validate_and_normalise_weights Reused by all test classes
    metrics.wauc Already supports sample_weight No change needed
    scipy.stats.permutation_test Statistic closure captures w from outer scope Labels permuted; weights are fixed

    Weight Propagation in Permutation Test¶

    scipy.stats.permutation_test calls statistic(x, y) where x and y are permuted slices of the pooled data. The statistic closure must reconstruct sample_weight per group from the original weight array using the same pooling logic as concat_samples / assign_labels:

    # Conceptual sketch — not final implementation
n_a = len(group_a_scores)
w_a = sample_weight[:n_a]   # weights for group A stay with group A's positions
w_b = sample_weight[n_a:]   # weights for group B
pooled_w = np.concatenate([w_a, w_b])

def statistic(a_perm, b_perm):
    actual_perm = assign_labels([a_perm, b_perm])
    predicted_perm = concat_samples([a_perm, b_perm])
    return metric(actual_perm, predicted_perm, sample_weight=pooled_w)

    Security & Privacy¶

    • sample_weight values are numerical arrays used only in internal arithmetic; no logging, serialisation, or external transmission.
    • Input validation (validate_and_normalise_weights) prevents NaN/inf weights from silently corrupting results.

    4. Risks & Roadmap¶

    Phased Rollout¶

    MVP (this release) - sample_weight=None kwarg on CTST and WeightedAUC - Full-pipeline propagation: permutation test + Bayesian bootstrap - validate_and_normalise_weights utility - Backwards-compatibility regression tests: equal weights → identical results to current implementation - Documentation: update docstrings for CTST, WeightedAUC, _bayesboot.bayesian_posterior

    v1.1 - Example notebook/docs page demonstrating all three weighting scenarios on the credit dataset - Add sample_weight to CTST.from_samples convenience constructor

    v2.0 - Consider shipping an optional density_ratio extras group with a convenience helper (classifier-based density ratio estimator) behind an import guard — keeps core dependency-free

    Technical Risks¶

    Risk Likelihood Impact Mitigation
    Permutation test label/weight misalignment Medium High — silently wrong p-values Pin a regression test that checks weight assignment under permutation with a crafted asymmetric weight array
    Dirichlet × user-weight interaction produces near-zero combined weights Medium Medium — numerical instability in wauc Clip combined weights at eps before normalisation; add test with extreme weight vectors
    Downstream callers pass un-normalised weights and misinterpret results High Low — results are valid but scaled differently Internal normalisation to sum = n; document convention clearly
    scipy.stats.permutation_test API change breaks the closure pattern Low High Pin scipy >= 1.8 (current constraint) and add an integration test against the full permutation path
